To answer your question regarding the custom Cypher engine:
The custom engine in `src/cypher/cypher.c` essentially functions as a compiler that translates Cypher syntax into standard SQL queries, because SQLite does not natively understand Cypher. Since Gorgonzola is a native graph database that *does* understand Cypher natively, my initial thought was to remove the maintenance burden of the custom translator entirely and simply pass raw Cypher strings straight into Gorgonzola.

However, as you pointed out, if we are keeping the dual-database structure (where SQLite stays active), we absolutely **must** keep the custom Cypher engine. Otherwise, we'd lose the ability to query the SQLite backend with Cypher! 

The updated plan now proposes a **query routing mechanism**:
- We keep both databases running side-by-side.
- Data writes (nodes, edges) are mirrored to both.
- Query execution can be routed. Standard/legacy queries can run through the custom translator to hit SQLite, while advanced graph traversals (which SQLite struggles with) can be sent directly to Gorgonzola as raw strings. This also opens up the ability to A/B test their performance.

5: 
6: # Git Submodule Workflow
7: 
8: Use this workflow whenever you make changes inside a git submodule, or when you are asked to "update submodules" or "push submodules".
9: 
10: ## The Problem
11: When you commit changes inside a submodule but fail to push those commits to the submodule's remote, the parent repository will fail to push. Git blocks pushing the parent repository if it references submodule commits that do not exist on any remote.
12: 
13: ## The Workflow
14: 
15: When updating or committing to submodules, always follow these steps:
16: 
17: 1. **Commit Local Submodule Changes**: 
18:    Ensure you are in the submodule directory and commit the changes.
19:    `cd path/to/submodule && git add . && git commit -m "Your message"`
20: 
21: 2. **Push Submodule Commits**: 
22:    Push the submodule to its remote branch. Since submodules often default to a detached HEAD state, you must push explicitly using `HEAD:<branch_name>`. Find the correct default branch (e.g. `main` or `master`) by checking `git branch -a` or `git remote show origin`.
23:    `git push origin HEAD:main` or `git push origin HEAD:master`
24: 
25: 3. **Commit the Submodule Pointer in Parent**: 
26:    Go back to the parent repository, add the submodule directory, and commit the pointer update.
27:    `cd /path/to/parent`
28:    `git add path/to/submodule`
29:    `git commit -m "chore: update submodule pointer"`
30: 
31: 4. **Push the Parent Repository**: 
32:    Finally, push the parent repository.
33:    `git push`
34: 
35: ## Tips
36: - For deeply nested submodules (submodules inside submodules), apply this workflow from the deepest submodule upwards.
37: - You can check if the parent will fail to push by doing a dry-run or by reading the error: `The following submodule paths contain changes that can not be found on any remote`.

- We keep both databases running side-by-side.
- Data writes (nodes, edges) are mirrored to both.
- Query execution can be routed. Standard/legacy queries can run through the custom translator to hit SQLite, while advanced graph traversals (which SQLite struggles with) can be sent directly to Gorgonzola as raw strings. This also opens up the ability to A/B test their performance.

23:    `git push origin HEAD:main` or `git push origin HEAD:master`

29:    `git commit -m "chore: update submodule pointer"`
